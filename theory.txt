1. Приложение запущено в докер-контейнере через docker-compose. Контейнер рестартится, логов нет, так как не успевает писать. Что будешь делать?
Если через docker-compose, то проверить все services, убедиться в этом файле, что все параметры указаны правильно.
Можно попробовать посмотреть предыдущие логи имея id контейнера командой docker logs ID.
Можно настроить вывод логов в отдельный файл внутри контейнера docker logs id > /mylogs.log.

2. Деплой сервиса провалился с сообщением типа “при запуске контейнера возникла ошибка” без деталей. Твои действия?
Проверить логи через команду docker logs.
Проверить зависимости и необходимые настройки сервиса.
Попробовать запустить в -it режиме или debug режиме, если возможно.

3. Билд завершился сообщением “Репозиторий не найден“. Что будешь делать?
Проверить верно ли указана ссылка, существует ли репозиторий по ней. Является ли он открытым для использования (не "приватным").
Если для доступа к репозиторию необходима авторизация, проверить их правильность.
Проверить интернет/хост.

4. Есть система неважно какой структуры. Требуется настроить мониторинг с нуля. Какие уровни ты охватишь?
-Мониторинг серверов: их состояние, ресурсы.
-Мониторинг ОС: работа базовых служб и процессов, логирования.
-Мониторинг приложения: работа приложений, API, взаимодействия между БД.
-Мониторинг "интерфейсов": клиент-приложение и все что происходит на этом пути.
Далее, согласно теории и google, следует бизнес-мониторинги, направленные в меньше степени на техническое обеспечение и поддержку.
